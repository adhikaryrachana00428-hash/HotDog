Data Flow Summary

1
Request arrives at Vercel Edge Network, which checks the Clerk session token via middleware before the request ever touches the app server.
2
Next.js App Router renders Server Components on the server, fetching data directly from PostgreSQL via Prisma — no client roundtrip needed for initial data.
3
Redis sits between the API and the database, caching expensive queries and storing session state to keep response times under 100ms.
4
Server Actions handle mutations (form submissions, state updates) directly from Client Components without building explicit API endpoints.

Engineering Relationships

How every component communicates with the rest of the system — and why those connections matter.

Key Dependency Chains

→ Vercel Middleware → Clerk SDK → Session Cookie → Protected Page

Every request is intercepted at the edge before reaching Next.js. Clerk validates the session token — unauthenticated requests are redirected to the login page without ever touching application code.

Data Mutation Flow
UI Form → Server Action → Zod Validation → Prisma Client → PostgreSQL

Form submissions bypass traditional REST endpoints. Server Actions run on the server in response to a POST request, validated by Zod before any database write occurs — eliminating an entire class of injection vulnerabilities.

Cache Read Flow
API Request → Redis Check → Cache Miss? → PostgreSQL → Cache + Return

The API always checks Redis first. On a cache miss, it queries PostgreSQL, stores the result in Redis with a TTL, and returns the data. Subsequent identical requests return in microseconds instead of milliseconds.

Engineering Decisions

The reasoning behind each major technology choice — including the tradeoffs that were consciously accepted.

01

Chose Clerk over Auth.js (NextAuth)

High Impact Decision
✅ Advantages Gained
  • Zero auth infrastructure to maintain
  • Pre-built UI components (sign-in, sign-up, user profile)
  • Organizations and multi-tenancy out of the box
  • Webhook events for user lifecycle management
⚠️ Tradeoffs Accepted
  • External service dependency for all user sessions
  • Vendor lock-in makes future migration complex
  • Paid tier required for production at scale
02

App Router over Pages Router

Architectural Decision
✅ Advantages Gained
  • Server Components reduce client-side JavaScript
  • Nested layouts eliminate prop-drilling across routes
  • Streaming and Suspense for better perceived performance
  • Server Actions for type-safe form handling
⚠️ Tradeoffs Accepted
  • Steeper learning curve for the team
  • Fewer community examples vs. Pages Router
  • Some ecosystem libraries not yet compatible
03

Prisma as the Database Interface

Medium Impact Decision
✅ Advantages Gained
  • Auto-generated TypeScript types from schema
  • Schema migrations with version history
  • Readable query builder instead of raw SQL
  • Prisma Studio for visual database browsing
⚠️ Tradeoffs Accepted
  • N+1 query problem if relations aren't handled carefully
  • Cold start overhead in serverless environments
  • Complex queries sometimes need raw SQL fallback

Smart Recommendations

6 technology alternatives and architectural improvements identified specifically for this project.

Upgrade

Consider Drizzle ORM as a Prisma Alternative

Drizzle is a newer, lighter ORM with a SQL-like query builder and zero cold start overhead — a significant advantage in serverless environments where Prisma's connection pooling can cause latency spikes.

Medium migration effort ↑ Performance · ↓ Cold starts
Alternative

Better Auth instead of Clerk for Long-term Projects

Better Auth is a newer open-source auth library that runs entirely on your own infrastructure. It eliminates Clerk's vendor lock-in and cost at scale while providing comparable features including OAuth, sessions, and organization management.

High migration effort ↓ Cost · ↑ Data ownership
Add

Add Playwright for End-to-End Testing

The project currently has unit tests but no E2E coverage. Playwright integrates with Next.js natively and can test the full authentication flow, form submissions, and payment processes in a real browser environment.

Low implementation effort ↑ Confidence · ↓ Regression risk
Add

OpenTelemetry for Observability

The application has no distributed tracing. Adding OpenTelemetry with a provider like Axiom or Grafana Cloud would let you trace a single user request through Middleware → Server Action → Prisma → PostgreSQL → Redis.

Medium implementation effort ↑ Debuggability · ↑ Production visibility
Optimize

Replace Redis with Upstash for Serverless Compatibility

Traditional Redis requires a persistent connection, which is problematic in serverless functions. Upstash provides a REST-based Redis API that works perfectly with Vercel's serverless model and charges per request rather than per instance.

Low migration effort ↓ Cost · ↑ Serverless compatibility
Add

Turborepo for Monorepo Management

As the codebase grows, splitting into a Turborepo monorepo would allow sharing TypeScript types, Zod schemas, and utility functions between the web app and any future mobile or API-only services.

Medium implementation effort ↑ Scalability · ↑ Code reuse

npx prisma migrate dev
